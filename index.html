Connection Pool
Object Creational Pattern
Intent
Manage a pool of reusable objects to avoid expensive acquisition and release of resources. Provides thread-safe access to a limited number of resource instances, automatically handling their lifecycle through RAII semantics.
Also Known As
Resource Pool, Object Pool with Handle, RAII Pool Manager
Motivation
Database connections are expensive to create (100-300ms) and there's typically a limit on concurrent connections. Applications need efficient connection reuse without the complexity of manual resource management. Consider a web server handling thousands of requests per second - creating a new database connection for each request would be prohibitively expensive and could exhaust database resources.

The Connection Pool pattern solves this by maintaining a collection of pre-initialized connections that can be borrowed and returned. The Handle idiom ensures automatic return through RAII, preventing resource leaks even in the presence of exceptions.
Applicability
Use the Connection Pool pattern when:
  • Object creation is expensive (network connections, thread creation, large buffers)
  • You need to limit the number of instances due to resource constraints
  • Objects are frequently acquired and released
  • Thread-safety is required for concurrent access
  • Automatic resource management is desired (RAII)
Structure
Participants
ConnectionPool
  • Maintains pool of reusable connections
  • Provides thread-safe access through mutex
  • Tracks available and in-use connections
  • Creates ConnectionHandle for RAII management
ConnectionHandle
  • RAII wrapper for borrowed connection
  • Provides transparent access via operator overloading
  • Automatically returns connection on destruction
  • Move-only semantics (no copying)
PooledConnection
  • Wraps actual connection with metadata
  • Tracks usage state and timestamp
Client
  • Requests connections from pool
  • Uses connection through handle interface
  • Automatic cleanup via RAII
Collaborations
Consequences
Benefits:
  • Performance: Eliminates expensive connection creation/destruction (100x+ speedup)
  • Resource Control: Enforces connection limits, prevents resource exhaustion
  • Thread Safety: Built-in synchronization for concurrent access
  • Automatic Management: RAII ensures no resource leaks
  • Transparency: Handle provides seamless access through operator overloading
Liabilities:
  • Serialization Overhead: Mutex creates contention point (microseconds per acquisition)
  • Memory Overhead: Maintains minimum connections even when idle
  • Complexity: More complex than direct connection creation
  • Single-Thread Inefficiency: Thread-safety overhead without benefit in single-threaded apps
Implementation
class ConnectionPool {
private:
    struct PooledConnection {
        std::unique_ptr<pqxx::connection> conn;
        std::chrono::steady_clock::time_point last_used;
        bool in_use;
    };
    
    std::vector<PooledConnection> connections;
    std::queue<size_t> available_indices;
    mutable std::mutex pool_mutex;
    std::condition_variable pool_cv;
    
public:
    class ConnectionHandle {
    private:
        pqxx::connection* conn;
        ConnectionPool* pool;
        size_t index;
        
    public:
        ~ConnectionHandle() {
            if (pool && conn) {
                pool->returnConnection(index);
            }
        }
        
        pqxx::connection& operator*() { return *conn; }
        pqxx::connection* operator->() { return conn; }
        
        // Move-only semantics
        ConnectionHandle(ConnectionHandle&& other) noexcept;
        ConnectionHandle(const ConnectionHandle&) = delete;
    };
    
    ConnectionHandle getConnection() {
        std::unique_lock<std::mutex> lock(pool_mutex);
        
        // Wait if no connections available
        pool_cv.wait(lock, [this] { 
            return !available_indices.empty(); 
        });
        
        size_t index = available_indices.front();
        available_indices.pop();
        connections[index].in_use = true;
        
        return ConnectionHandle(
            connections[index].conn.get(), 
            this, 
            index
        );
    }
    
private:
    void returnConnection(size_t index) {
        std::lock_guard<std::mutex> lock(pool_mutex);
        connections[index].in_use = false;
        available_indices.push(index);
        pool_cv.notify_one();
    }
};
Sample Code
// Usage example
void processRequests(ConnectionPool& pool) {
    std::vector<std::thread> threads;
    
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back([&pool, i]() {
            // Acquire connection (automatic via RAII)
            auto handle = pool.getConnection();
            
            // Use connection transparently
            pqxx::work txn(*handle);
            auto result = txn.exec(
                "SELECT * FROM users WHERE id = " + 
                std::to_string(i)
            );
            txn.commit();
            
            // Connection automatically returned when 
            // handle goes out of scope
        }); // ~ConnectionHandle() called here
    }
    
    for (auto& t : threads) {
        t.join();
    }
}

Related Patterns
  • Singleton: Pool itself is often implemented as a singleton
  • Factory Method: Pool acts as factory for connection handles
  • Proxy: ConnectionHandle proxies the actual connection
  • Flyweight: Shares immutable connection configuration
  • Object Pool: Base pattern that Connection Pool specializes
